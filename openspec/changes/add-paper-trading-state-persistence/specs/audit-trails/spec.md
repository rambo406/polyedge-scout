## ADDED Requirements

### Requirement: All state mutations are recorded in an immutable audit log
The system SHALL record every state mutation (trade creation, trade settlement, bankroll change, state recovery) as one or more entries in the `AuditLog` table, following ISO 27001 and ISO 9001 audit trail requirements.

#### Scenario: Trade creation generates audit entries
- **WHEN** `ExecutePaperTrade` successfully creates a new trade
- **THEN** the system SHALL insert an `AuditLogEntry` with `EntityType = "Trade"`, `EntityId = <trade.Id>`, `Action = Created`, `UserId = "system"`, and a new `CorrelationId`
- **AND** the system SHALL insert an `AuditLogEntry` with `EntityType = "AppState"`, `EntityId = "Bankroll"`, `Action = Updated`, `PropertyName = "Bankroll"`, `OldValue = <previous bankroll>`, `NewValue = <new bankroll>`, and the same `CorrelationId`

#### Scenario: Trade settlement generates audit entries
- **WHEN** `SettleTrade` successfully settles a trade
- **THEN** the system SHALL insert an `AuditLogEntry` with `EntityType = "Trade"`, `EntityId = <trade.Id>`, `Action = Settled`, `PropertyName = "Status"`, `OldValue = "Open"`, `NewValue = "Settled"`, and a new `CorrelationId`
- **AND** the system SHALL insert an `AuditLogEntry` with `EntityType = "TradeResult"`, `EntityId = <tradeResult.Id>`, `Action = Created`, and the same `CorrelationId`
- **AND** the system SHALL insert an `AuditLogEntry` with `EntityType = "AppState"`, `EntityId = "Bankroll"`, `Action = Updated`, `PropertyName = "Bankroll"`, `OldValue = <previous bankroll>`, `NewValue = <new bankroll>`, and the same `CorrelationId`

#### Scenario: State recovery generates an audit entry
- **WHEN** the application recovers state from the database on startup
- **THEN** the system SHALL insert an `AuditLogEntry` with `EntityType = "AppState"`, `Action = Recovered`, `NewValue` containing a summary of recovered state (trade counts, bankroll)

### Requirement: Audit log timestamps use ISO 8601 UTC format
The system SHALL store all audit log timestamps in ISO 8601 UTC format to ensure unambiguous temporal ordering.

#### Scenario: Timestamps are stored in ISO 8601 format
- **WHEN** an `AuditLogEntry` is created
- **THEN** the `Timestamp` field SHALL be in ISO 8601 UTC format, e.g. `"2026-03-23T14:30:00.0000000Z"`
- **AND** the timestamp SHALL use `DateTime.UtcNow` (or an injected clock) at the moment the audit entry is created

#### Scenario: Timestamps enable temporal ordering
- **WHEN** audit log entries are queried by date range
- **THEN** the entries SHALL be orderable by `Timestamp` in chronological order
- **AND** the ordering SHALL be consistent regardless of time zone

### Requirement: Audit log is append-only (immutable)
The system SHALL enforce that the audit log is append-only. No UPDATE or DELETE operations SHALL be performed on the `AuditLog` table through the application layer.

#### Scenario: IAuditLogRepository only exposes add and query methods
- **WHEN** the `IAuditLogRepository` interface is defined
- **THEN** it SHALL expose `AddAsync`, `AddRangeAsync`, and query methods (`GetByEntityAsync`, `GetByCorrelationIdAsync`, `GetByDateRangeAsync`)
- **AND** it SHALL NOT expose any `UpdateAsync`, `DeleteAsync`, or `RemoveAsync` methods

#### Scenario: Audit entries cannot be modified after creation
- **WHEN** an `AuditLogEntry` has been persisted to the database
- **THEN** there SHALL be no application-layer mechanism to modify or delete it
- **AND** the `AuditLogEntry` entity type SHALL NOT be tracked for updates by the DbContext (or updates SHALL be explicitly blocked)

### Requirement: Correlation IDs group related audit entries
The system SHALL assign a shared `CorrelationId` to all audit entries generated by a single business operation, enabling reconstruction of complete operation histories.

#### Scenario: Trade execution audit entries share a correlation ID
- **WHEN** `ExecutePaperTrade` generates audit entries for the trade creation and bankroll update
- **THEN** all audit entries from that operation SHALL have the same `CorrelationId` (a new GUID)

#### Scenario: Trade settlement audit entries share a correlation ID
- **WHEN** `SettleTrade` generates audit entries for the trade status change, trade result creation, and bankroll update
- **THEN** all audit entries from that operation SHALL have the same `CorrelationId` (a new GUID)

#### Scenario: Querying by correlation ID returns all related entries
- **WHEN** `IAuditLogRepository.GetByCorrelationIdAsync` is called with a correlation ID
- **THEN** the system SHALL return all audit log entries with that correlation ID

### Requirement: Audit entries and data changes are atomically committed
The system SHALL write audit log entries within the same database transaction as the data changes they describe, ensuring consistency between the audit trail and the system state.

#### Scenario: Audit entries are committed with data changes
- **WHEN** a state mutation (trade execution or settlement) is committed
- **THEN** the audit log entries for that operation SHALL be committed in the same transaction

#### Scenario: Audit entries are rolled back with data changes
- **WHEN** a state mutation transaction fails and is rolled back
- **THEN** the audit log entries for that operation SHALL also be rolled back
- **AND** the audit log SHALL NOT contain entries for operations that did not complete

### Requirement: Audit log supports entity-level queries
The system SHALL support querying the audit log by entity type and entity ID to enable full audit trail reconstruction for any tracked entity.

#### Scenario: Query audit history for a specific trade
- **WHEN** `IAuditLogRepository.GetByEntityAsync("Trade", <tradeId>)` is called
- **THEN** the system SHALL return all audit log entries for that trade in chronological order (Created, then potentially Settled)

#### Scenario: Query audit history for bankroll changes
- **WHEN** `IAuditLogRepository.GetByEntityAsync("AppState", "Bankroll")` is called
- **THEN** the system SHALL return all audit log entries for bankroll changes in chronological order, each with `OldValue` and `NewValue` showing the before/after bankroll amounts

#### Scenario: Query audit log by date range
- **WHEN** `IAuditLogRepository.GetByDateRangeAsync(from, to)` is called
- **THEN** the system SHALL return all audit log entries with `Timestamp` between `from` and `to` (inclusive) in chronological order

### Requirement: ISO 27001 compliance — event logging and protection
The audit trail SHALL comply with ISO 27001 Annex A controls for event logging (A.12.4.1) and protection of log information (A.12.4.2).

#### Scenario: All security-relevant events are logged (A.12.4.1)
- **WHEN** any state change occurs in the system (trade creation, settlement, bankroll modification, state recovery)
- **THEN** the event SHALL be recorded with: timestamp (UTC), actor identification (`UserId`), action performed, affected entity, and outcome details (before/after values where applicable)

#### Scenario: Log integrity is protected (A.12.4.2)
- **WHEN** audit log entries exist in the database
- **THEN** the application layer SHALL NOT provide any mechanism to modify or delete them
- **AND** the `IAuditLogRepository` SHALL enforce append-only semantics
- **AND** any attempt to modify audit records through EF Core change tracking SHALL be prevented or blocked

### Requirement: ISO 9001 compliance — documented information and traceability
The audit trail SHALL comply with ISO 9001 requirements for documented information (7.5) and provide traceability for corrective action analysis (10.2).

#### Scenario: Full traceability of trade lifecycle (7.5)
- **WHEN** a trade is created and later settled
- **THEN** the audit log SHALL contain a complete, chronological record of all state changes for that trade
- **AND** the audit trail SHALL enable reconstruction of the exact system state at any point in time by replaying the log entries

#### Scenario: Root cause analysis support (10.2)
- **WHEN** an unexpected trading outcome occurs (e.g., unexpected P&L)
- **THEN** the audit log SHALL contain sufficient detail (before/after values, timestamps, correlation IDs) to trace the exact sequence of events that led to the outcome
